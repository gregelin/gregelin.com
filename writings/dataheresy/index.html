Introduction

Relational database technology is a 30-year old technology. It's very good. It's so good, we have tended it use it as a one-size-fits-all-solution. And that is a problem. As more information goes digital, we try to shoe horn into relational databases all sorts of structured information that does not necessarily conform to the nice neat transactions and fixed, long-term data structures of relational databases. Also, RDMS were not designed to deal with Internet-scale data.

What follows are what many might consider pretty heretical thoughts for working with data. But like other heresies, many individuals will agree with these if you ask privately one-on-one.

Data Heresies
  • The experience of sharing data within an enterprise or between enterprises is painful. Vendors will make this better, but they are likely to be startups and web 2.0 companies that have been forced to entirely new solutions for dealing with data, like BigTable, Hadoop, MapReduce, and Memcached. In the future, there will be players as enterprise data moves to the cloud and the cloud, virtualizations, and clusters moves into the enterprise. Traditional database vendors role is likely to diminish unless they expand their thinking.
  • ETL (Extraction Transformation Loading) and Data Quality tools are crappy, cumbersome, and lack standardization, especially when compared to content production and dissemination. Where there are good ETL tools, they are incredibly expensive. Also, most if not all ETL tools are ignorant of how to leverage the web for data quality.
  • Future of data sharing is in versioning systems and "data containers". I'm starting to treat data more like code and exploring how to use tools like github to manage data I want to share. (Data loaded into a database should be treated as a "build" instead of a system of record; the non-proprietary data dump should be system of record.) To see what is happening in versioning systems, see GitHub or playback feature of Google Waves (about 18-20min in video). To see "data containers" look at Socrata.com or Microsoft's Open Govt Data Initiative. These are small data sets, but the ideas are especially applicable to big data in the clouds, namely, getting data as some type of container or image with not just data but business rules, metadata, provenance, etc.
  • We need tools for "semantic scaling" data, especially large data streams produced by sensors, science, or life/business streams.
